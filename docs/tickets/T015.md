# T015 - Prompt control plane and plugin architecture

## Problem / Goal

- Prompt text, schema definitions, and output parsing are split across multiple modules and files.
- This makes it hard to reason about behavior changes and difficult for users to customize safely.
- Introduce a single prompt control plane that still supports the three existing process flows: `digest`, `merge`, and `report`.
- Allow user-defined TypeScript plugins to customize prompting and schema behavior in one place.

## Proposed Change

- Build a centralized prompt registry that defines operation contracts for `digest`, `merge`, and `report`.
- Move operation-level prompt construction, response schema, and parser wiring into that registry.
- Add a plugin API and config loader so users can patch/override operation behavior via `proma.config.ts`.
- Execute AI calls through a unified runtime executor that consumes the registry.
- Remove the legacy markdown prompt-template mechanism (`prompts/*` + `src/prompt-template.ts`) in favor of code-first definitions.

## Architecture Plan

- Add a new `src/prompting/` package with focused modules:
  - `types.ts` for operation/registry/plugin contracts
  - `registry.ts` for built-in operation definitions
  - `plugin.ts` for plugin API contracts
  - `load.ts` for config and plugin composition
  - `validate.ts` for composed registry startup checks
  - `execute.ts` for model execution + parser handoff
- Keep domain parsing logic in existing modules (`src/digest.ts`, `src/report.ts`) but invoke it through registry definitions.
- Thread the loaded registry through CLI command handlers once per process startup.

## Delivery Plan

- Phase 1: Define typed contracts and built-in registry (`T015.1`, `T015.2`).
- Phase 2: Add plugin/config loader and registry validation (`T015.3`, `T015.4`).
- Phase 3: Integrate runtime in `digest`, `merge`, `report` command paths (`T015.5`).
- Phase 4: Remove legacy prompt template mechanism and clean tests/docs (`T015.6`, `T015.7`, `T015.8`).

## Scope

In scope:

- New typed prompting API surface.
- Built-in operation registry for the three existing flows.
- Plugin/config loading, validation, and error handling.
- CLI and generation pipeline integration.
- Tests and docs updates.

Out of scope:

- Adding new process types beyond `digest`, `merge`, and `report`.
- Cross-repo plugin marketplace or remote plugin loading.

## Non-goals

- Backward compatibility with existing prompt markdown files.
- UI-level plugin management.

## Implementation Notes

- Keep strict structured outputs (`json_schema`) for all three operations.
- Registry validation should fail fast at startup when plugins produce invalid operation definitions.
- Plugin order should be deterministic; later plugins can override earlier ones.
- v1 plugin constraints:
  - Plugins can mutate only existing operations (`digest`, `merge`, `report`).
  - Plugins cannot register new operation kinds.
  - Plugin hooks are synchronous/async setup hooks only (no background workers).
- Error handling contract:
  - Config load errors are surfaced before command execution.
  - Validation errors include operation key and failing contract field.
  - Runtime parser failures preserve current user-facing error quality.

## Acceptance Criteria

- [ ] A single registry defines prompting, schema, and parser contracts for `digest`, `merge`, and `report`.
- [ ] Users can customize operation behavior via TypeScript plugins in `proma.config.ts`.
- [ ] CLI uses the unified runtime for all three process flows.
- [ ] Legacy prompt markdown/template files are no longer required for runtime behavior.
- [ ] Tests validate built-in behavior and plugin overrides.
- [ ] Documentation explains the new architecture and plugin usage.
- [ ] Regression tests confirm behavior parity for default `digest`, `merge`, and `report` outputs.
