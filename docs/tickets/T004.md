# T004 - Two-stage digest organization with topic merging

## Problem / Goal

- The current digest flow writes output directly into category folders, which fragments related notes across many date-based files.
- We need a staged pipeline that first captures raw digest outputs, then groups and merges them into topic files for better organization.
- Topic routing should use AI with awareness of existing topic files.
- The merge step must be user-confirmed with a diff preview before file changes are applied.

## Proposed Change

- Implement a 3-stage processing flow for each digest item.
- Stage 1: write raw digest markdown to `<project>/notes/<category>_<YYYY-MM-DD>_<index>.md`.
- Stage 2: select one or more topic targets using AI by comparing the digest item against existing topic files in `<project>/<category>/`.
- Stage 2 must provide candidate topic filenames/slugs and metadata to AI so it can choose existing file(s) or request creation of a new topic file.
- New topic files are created at `<project>/<category>/<short-description>.md`, where `<short-description>` is normalized to a filesystem-safe kebab-case slug.
- Stage 3: generate merged content per target and show diff preview; apply merge only after explicit user confirmation.

## Scope

In scope:

- Add Stage 1 raw-output writer under `<project>/notes/` with category/date/index naming.
- Add topic discovery in `<project>/<category>/` and candidate extraction for AI routing.
- Add AI schema/prompt for selecting one or more existing topic file(s) or `create_new`.
- Add topic file creation with YAML front matter metadata.
- Add merge planner that keeps topic files canonical (`Summary/Key Points/References`) rather than append-only entry logs.
- Add interactive diff confirmation before writing merged content.
- Update CLI output/messages and tests to cover staged flow.
- Update README to document staged behavior and confirmation step.

Out of scope:

- Semantic deduplication across all projects/categories.
- Background daemon or async queue processing.
- Non-interactive auto-merge mode that bypasses confirmation.

## Non-goals

- Replacing category taxonomy (`planning | research | discussion`).
- Building a full note graph or backlink system.

## Implementation Notes

- Keep existing digest generation intact; change persistence/orchestration path.
- Introduce dedicated functions for:
  - raw note write allocation in `<project>/notes`
  - topic candidate discovery in `<project>/<category>/*.md`
  - topic selection/create decision via AI structured output
  - merge diff generation + confirmation
  - final merge write
- Normalize AI-proposed short descriptions to kebab-case slugs.
- Topic files use front matter metadata such as:
  - `topic`
  - `category`
  - `created_at`
  - `updated_at`
  - `tags`
  - `sources`
-  `source_refs`
-  `merged_digest_ids`
- Refresh `updated_at` on each confirmed merge and merge tags/sources deterministically.
- Normalize `tags` as lowercase kebab-case strings with de-duplication and sorted output.
- Canonical merge behavior must deduplicate repeated ingestion and avoid appending redundant sections.
- Diff preview should support per-target confirmation when AI selects multiple files.
- If user rejects a merge target, keep Stage 1 raw note and skip topic write for that target.

## Acceptance Criteria

- [x] Running `bun run index.ts digest --input <file> --project <project>` writes Stage 1 files to `<project>/notes/<category>_<YYYY-MM-DD>_<index>.md`.
- [x] Stage 2 sends existing topic candidates (slug + metadata) to AI for routing decisions.
- [x] Stage 2 supports selecting multiple existing topic files and/or creating a new topic file.
- [x] New topic files are created with required YAML front matter metadata.
- [x] Stage 3 shows a diff preview before each merge and writes only after explicit user confirmation.
- [x] Rejected merge targets do not modify topic files, and Stage 1 raw files remain.
- [x] Confirmed merges update front matter metadata (`updated_at`, merged `tags`/`sources`, `source_refs`, `merged_digest_ids`) correctly.
- [x] Topic files keep one canonical merged view and do not append duplicate digest entry blocks.
- [x] Re-ingesting a digest with already merged references is treated as no-op for topic content.
- [x] Tests cover raw file naming/indexing, topic selection/creation, multi-target confirmation-gated merge writes, and README documentation.
