# T016.1 - Make `proma` installable as a CLI and importable module

## Problem / Goal

- Current project metadata is not publishable/consumable (`private` package, no binary mapping, CLI side effects at root import).
- External repositories need a stable `proma` executable while retaining programmatic import options.

## Proposed Change

- Update `package.json` to publish-friendly metadata (versioned package with CLI `bin` mapping and explicit entry/export strategy).
- Introduce a dedicated CLI executable entrypoint with shebang that delegates to `runCli`.
- Keep root module export surface library-safe (no immediate process execution on import).

## Scope

In scope:

- Metadata and file layout changes needed to run `proma` after install.
- Entry wiring for command execution and import safety.

Out of scope:

- Publishing to npm within this ticket.

## Implementation Notes

- Preserve existing command behavior and exit code semantics.
- Ensure Bun-first workflows remain documented and functional.
- Avoid introducing breaking API renames for existing internal imports.

## Acceptance Criteria

- [ ] Package metadata is valid for dependency installation in another repository.
- [ ] Installing package exposes a runnable `proma` command.
- [ ] Importing package does not trigger CLI execution side effects.
- [ ] Existing CLI tests continue to pass with new entrypoint layout.
