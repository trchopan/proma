# T023.3 - Semantic merge for canonical topic content

## Problem / Goal

- Deterministic union-merge cannot distinguish semantically unrelated content, so unrelated points can leak into a topic.
- We need semantic filtering while preserving robust behavior when AI output is invalid/unavailable.

## Proposed Change

- Add a new built-in prompt operation (for example `merge_content`) with strict structured output for canonical topic sections.
- Feed existing canonical topic content + incoming digest item into semantic merge generation.
- Integrate semantic output into `buildTopicMergeContent` flow with strict validation.
- If semantic merge fails (parse/refusal/runtime), fall back to existing deterministic merge logic.

## Scope

In scope:

- Prompt operation type additions and validation support.
- Built-in prompt definition and parse logic for semantic canonical output.
- Topic merge service wiring to call semantic merge operation.
- Fallback path and logging hooks for failures.
- Tests for semantic success path and deterministic fallback path.

Out of scope:

- Replacing all deterministic merge utilities.
- Changing canonical markdown section structure.

## Implementation Notes

- Schema must keep timeline entry format compatible with existing canonical parsing rules.
- Relevance guardrails should explicitly prevent unrelated schedule/task details from entering policy-only topics.
- Preserve idempotency/no-change detection semantics.

## Acceptance Criteria

- [ ] New semantic merge prompt operation exists and passes registry validation.
- [ ] Topic merge service consumes semantic canonical output when valid.
- [ ] Invalid/refused semantic output triggers deterministic fallback without crashing merge.
- [ ] Idempotent/no-change behavior remains intact.
- [ ] Service tests cover semantic merge filtering and fallback behavior.
