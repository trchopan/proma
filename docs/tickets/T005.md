# T005 - Split digest and merge scripts with staged merge state

## Problem / Goal

- The current `digest` command executes stage 1 (write notes) and stage 2/3 (topic routing + merge) in one run.
- We need separate scripts aligned to stage boundaries: first `digest` notes, then `merge`.
- Staged notes need persisted merge state so `merge-topic` only picks notes not yet merged.

## Proposed Change

- Keep `digest` as stage-1-only note generation.
- Add `merge` command that reads staged notes and performs topic routing + merge confirmation.
- Add YAML front matter to stage-1 note files with `merged: boolean`, defaulting to `false`.
- During `merge`, process only notes where `merged` is not `true`.
- After successful merge processing for a note, update front matter `merged: true`.

## Scope

In scope:

- CLI command split and argument parsing updates in `src/cli.ts`.
- Stage note front matter write/read support in `src/files.ts`.
- Pending-note discovery for `<project>/notes/*.md`.
- Merge-state update (`merged: true`) after successful processing.
- Tests for command split, pending-note filtering, and merge-state transitions.
- README command/flow documentation updates.

Out of scope:

- Reworking topic merge semantics beyond command split and staged note state.
- Background workers or non-interactive bulk merge mode.

## Non-goals

- Changing digest categorization taxonomy.
- Changing AI schema for digest item generation beyond what is needed to support split execution.

## Implementation Notes

- Preserve existing topic routing and diff confirmation logic; move orchestration from `digest` path to `merge`.
- Stage note files must remain parseable across runs; include enough metadata to reconstruct `DigestItem`.
- Treat missing `merged` as pending (`false`) for backward compatibility with existing notes.
- Keep per-target confirmation and skip behavior unchanged.

## Acceptance Criteria

- [ ] `bun run index.ts digest --input <file> --project <project>` writes stage-1 notes only and does not perform topic merges.
- [ ] Stage-1 notes include front matter with `merged: false`.
- [ ] `bun run index.ts merge --project <project>` processes only notes with `merged != true`.
- [ ] Successfully processed notes are updated to `merged: true`.
- [ ] Notes with rejected/skipped merges remain `merged: false`.
- [ ] Existing notes without `merged` are treated as pending and can be merged.
- [ ] Tests cover split commands, pending-note selection, and merge-state updates.
- [ ] README documents the two-command workflow.
