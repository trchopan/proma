# T011.5 - Config and error model cleanup

## Problem / Goal

- Env/defaults and error semantics are scattered, making failure handling inconsistent and harder to reason about.

## Proposed Change

- Add central config resolver (defaults + env validation).
- Introduce typed error classes (validation, parsing, I/O, AI transport/schema).
- Map typed errors to clearer CLI output while preserving exit code behavior.

## Scope

In scope:

- New config/error modules and adoption in key paths.
- Tests for mapping and error message consistency.

Out of scope:

- New CLI UX features or retries/backoff policies.

## Acceptance Criteria

- [ ] Runtime defaults/env access are centralized.
- [ ] Error categories are explicit and consistently handled.
- [ ] CLI still returns expected non-zero behavior on failures.
