# T017.5 - Add in-code API documentation for plugin/prompting APIs

## Problem / Goal

- External and internal developers can use plugin APIs, but key behavior details currently live mostly in markdown docs.
- We need source-level API documentation so Language Server hover/signature help shows correct usage and constraints where developers write code.

## Proposed Change

- Add TSDoc/JSDoc comments to exported plugin/prompting APIs under `src/prompting/`.
- Ensure comments are concise, accurate, and aligned with runtime behavior and existing tests.

## Scope

In scope:

- `src/prompting/plugin.ts`
  - `PromptPluginApi`
  - `PromptPlugin`
  - `PromaConfig`
- `src/prompting/load.ts`
  - `loadPromptRegistry`
- `src/prompting/types.ts`
  - core exported plugin/prompting contract types used by plugin authors
- `src/prompting/validate.ts`
  - `validatePromptRegistry`

Out of scope:

- CLI/storage APIs outside `src/prompting/`.
- Runtime logic changes to plugin loading, execution, or validation.

## Implementation Notes

- Use hover-friendly comments with clear intent, parameter semantics, and return behavior.
- Document deterministic config discovery order and plugin declaration-order execution where relevant.
- Include at least one minimal example snippet for plugin authoring on the primary API surface.
- Keep terminology consistent with `PLUGIN_SYSTEM.md` and source type names.

## Acceptance Criteria

- [ ] Exported plugin/prompting APIs in scope include Language Server-visible docs.
- [ ] Docs explain key behavior constraints (config search order, plugin ordering, contract validation expectations).
- [ ] At least one plugin authoring example is included in source comments.
- [ ] Documentation updates do not change runtime behavior.
- [ ] `bun run typecheck` passes.
- [ ] `bun test tests/prompting.test.ts` passes.
