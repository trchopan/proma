# T029 - Reorganize `src` into domain-layered structure (Option 2)

## Problem / Goal

- Logic for `digest`, `merge`, and `report` is currently split across mixed technical folders (`digest`, `services`, `storage`, and root files), making ownership boundaries unclear and onboarding slower.
- `src/cli.ts` orchestrates all command flows and imports many modules directly, which makes cross-domain change impact hard to reason about.
- `src/files.ts` is a storage facade, but merge/report behavior still spans `src/services/topic-merge.ts`, `src/storage/topic-files.ts`, `src/report.ts`, and `src/storage/report-files.ts`.
- Goal: adopt Option 2 (domain-layered structure) so code is discoverable by business domain and technical layer, while preserving behavior and CLI contracts.

## Review Findings (reasoning for refactor)

- **Cross-folder merge flow:** merge functionality is split across `src/digest/generate.ts` (AI generation APIs), `src/services/topic-merge.ts` (deterministic merge rules), and `src/storage/topic-files.ts` (topic file read/write + ranking).
- **Split report ownership:** report generation and parsing live in `src/report.ts`, while report file resolution/context/output live in `src/storage/report-files.ts`.
- **Mixed naming conventions:** directory names are currently mixed between domain (`digest`, `import`) and technical role (`services`, `storage`), which creates ambiguity when adding new features.
- **High import fan-in from CLI:** command orchestration in `src/cli.ts` depends on many modules from unrelated folders, increasing the cost of path navigation and refactor safety.
- **Existing stable seams we can preserve:** `src/files.ts` already centralizes storage exports and can be retained as a compatibility seam during migration.

## Refactor Decision

- Choose **Option 2 (domain + layer)** over pure feature folders because this codebase already has clear technical layers (CLI orchestration, storage adapters, integrations, core utilities).
- This approach improves discoverability without forcing behavior or ownership rewrites in the same ticket.
- It also allows incremental migration with lower risk: move paths first, preserve APIs, then optionally refine internals in follow-up tickets.

## Proposed Change

- Reorganize `src` into the following high-level layout:
  - `src/domain/{digest,merge,report}`
  - `src/storage/{digest,merge,report,import}`
  - `src/integrations/{ai,github,mcp}`
  - `src/core/{config,errors,logging,markdown,prompting}`
  - `src/cli`
- Update all imports/re-exports to match new paths.
- Keep CLI commands and runtime behavior unchanged.

Proposed path mapping (initial baseline):

- `src/digest/*` -> `src/domain/digest/*`
- `src/report.ts` -> `src/domain/report/report.ts`
- `src/services/topic-merge.ts` -> `src/domain/merge/topic-merge.ts`
- `src/storage/digest-notes.ts` -> `src/storage/digest/digest-notes.ts`
- `src/storage/topic-files.ts` -> `src/storage/merge/topic-files.ts`
- `src/storage/report-files.ts` -> `src/storage/report/report-files.ts`
- `src/storage/import-files.ts` -> `src/storage/import/import-files.ts`
- `src/ai/openai.ts` -> `src/integrations/ai/openai.ts`
- `src/import/github-client.ts` -> `src/integrations/github/client.ts`
- `src/import/mcp-client.ts` -> `src/integrations/mcp/client.ts`
- `src/import/transform.ts` -> `src/integrations/mcp/transform.ts`
- `src/config.ts` -> `src/core/config.ts`
- `src/errors.ts` -> `src/core/errors.ts`
- `src/logging.ts` -> `src/core/logging.ts`
- `src/markdown/*` -> `src/core/markdown/*`
- `src/prompting/*` -> `src/core/prompting/*`
- `src/files.ts` -> keep at `src/files.ts` in this ticket (compatibility facade), update internal exports to new storage paths.

## Scope

In scope:

- File/folder moves in `src` to match Option 2.
- Import path and barrel export updates required by the move.
- Test and CLI smoke verification after refactor.

Out of scope:

- Feature changes in digest/merge/report/import.
- Prompt/schema contract changes.
- Behavior rewrites beyond structural refactoring.

## Non-goals

- Introducing new command flags or CLI UX changes.
- Redesigning internal algorithms during this ticket.
- Splitting mixed-concern files into smaller modules (for example, ranking extraction from storage modules) unless required for import resolution.

## Implementation Notes

- Migrate in phases to keep the branch stable:
  1. Move `core` and `integrations` modules.
  2. Move `domain` modules (`digest`, `merge`, `report`).
  3. Move `storage` modules into domain subfolders.
  4. Update `src/files.ts` facade exports.
  5. Update remaining imports and run tests.
- Prefer temporary compatibility re-exports only when needed to keep diff readable and avoid large all-at-once risk.
- Validate no circular dependencies are introduced across `domain`, `storage`, `integrations`, and `core`.
- Keep naming stable where possible (`topic-merge.ts`, `report-files.ts`, etc.) to ease git blame continuity.

## Risks / Mitigations

- **Risk:** large import churn can cause hidden runtime breakage.  
  **Mitigation:** phase-by-phase moves + test run after each phase.
- **Risk:** accidentally changing behavior in mixed files during relocation.  
  **Mitigation:** no logic edits except path/import updates; isolate any required logic touch in separate commit section.
- **Risk:** merge conflicts with active tickets touching `src/cli.ts` and storage modules.  
  **Mitigation:** keep refactor branch short-lived and rebase frequently.
