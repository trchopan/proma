# T016.2 - Add `proma.config.js` discovery for plugin loading

## Problem / Goal

- Plugin loading currently assumes `proma.config.ts`, but external users commonly keep runtime config in JavaScript.
- We need a frictionless setup where `proma.config.js` works out of the box in consuming repositories.

## Proposed Change

- Extend config discovery to check a deterministic list of supported config files (at minimum `.ts` and `.js`, optionally `.mjs`).
- Load the first existing match and apply plugins with existing validation and error semantics.
- Add tests that assert `.js` config plugins are discovered and executed.

## Scope

In scope:

- Config filename discovery logic and associated tests.
- Documentation touchpoints that mention supported config filenames.

Out of scope:

- Alternate config directories or environment-based config path overrides.

## Implementation Notes

- Keep plugin execution order unchanged (declaration order inside config file).
- Keep error messages actionable and include loaded config filename when possible.

## Acceptance Criteria

- [ ] Loader supports `proma.config.js` in addition to existing TypeScript config behavior.
- [ ] Missing config still falls back to built-in registry without failure.
- [ ] Invalid `.js` plugin entries fail with readable errors.
- [ ] Tests cover successful `.js` plugin patch behavior.
