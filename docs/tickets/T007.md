# T007 - Add debugging-first logging system with verbose AI traces

## Problem / Goal

- The CLI currently writes ad-hoc console output, which is not enough for troubleshooting multi-step digest and merge runs.
- AI call behavior is difficult to debug after the fact because requests/responses are not persisted in run logs.
- We need a consistent logging system that is easy to use by default, while allowing deep debugging when explicitly requested.

## Proposed Change

- Introduce a centralized logging module that writes structured logs to a default `logs/` directory.
- Add global `--verbose` support so heavy logs are enabled only when requested.
- Keep default console output focused on user progress (high-level milestones and result counts).
- In verbose mode, emit detailed debug events for CLI and AI workflows.
- Capture full AI prompt and full AI response text in verbose logs to support deep debugging.

## Scope

In scope:

- Add `src/logging.ts` with:
  - run-scoped logger creation
  - structured JSONL file output under `logs/`
  - leveled logging helpers (`progress`, `info`, `debug`, `error`)
  - verbose-gated debug behavior
- Update CLI argument parsing to support global `--verbose` for `digest` and `merge`.
- Wire logger into `runCli` flow to record:
  - command start/end
  - major step transitions
  - per-step durations where practical
  - error details
- Instrument OpenAI chat completion path to log request/response lifecycle and timing.
- In verbose mode, persist full AI prompt text and full response text for each call.
- Add or update tests for arg parsing and logging behavior.
- Update README with logging and `--verbose` usage.

Out of scope:

- Remote log shipping or observability platform integration.
- Log rotation, retention policy tooling, or compression.
- Redaction/sanitization of AI prompt/response payloads in verbose mode.

## Non-goals

- Replacing the existing user-facing progress semantics of digest/merge commands.
- Changing digest/topic business logic beyond adding instrumentation.

## Implementation Notes

- Prefer newline-delimited JSON (JSONL) for file logs to keep output machine-readable and grep-friendly.
- Default log root must be `logs/` relative to current working directory unless explicitly overridden in future work.
- Use one run file per invocation with stable naming (for example timestamp + command + pid).
- Console behavior:
  - default mode: progress only
  - verbose mode: include debug details
- AI instrumentation should include at minimum:
  - model
  - endpoint/base URL
  - request start timestamp
  - duration
  - success/failure status
  - full prompt text (verbose)
  - full response text (verbose)

## Acceptance Criteria

- [ ] Running `digest` or `merge` creates a structured log file under `logs/` by default.
- [ ] Without `--verbose`, console output remains progress-focused and does not print heavy debug details.
- [ ] With `--verbose`, console output includes heavy debug information.
- [ ] CLI argument parsing accepts `--verbose` for both commands.
- [ ] AI request lifecycle events are logged with timing and status.
- [ ] In verbose mode, logs include full AI prompt text and full AI response text.
- [ ] Existing digest/merge behavior remains functionally unchanged except for logging.
- [ ] Tests cover `--verbose` parsing and at least one verbose AI logging path.
- [ ] README documents log location and verbose mode behavior.

## Risks / Considerations

- Verbose logs may contain sensitive information because full prompt/response content is intentionally captured.
- Log files may grow quickly for large inputs or long responses; monitor disk usage during heavy runs.
