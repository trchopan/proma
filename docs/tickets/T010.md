# T010 - Timeline section with strict ISO date and context format

## Problem / Goal

- Current digest output and canonical topic files only include `## Summary`, `## Key Points`, and `## References`.
- Teams need date-aware milestones in each item, and they need consistent machine-readable formatting.
- We need a new required `## Timeline` section where each entry includes a strict ISO date (`YYYY-MM-DD`) and short context so timeline data is both machine-validated and human-meaningful.

## Proposed Change

- Extend digest item schema with an optional `timeline` array of strings.
- Require every `timeline` entry to include both:
  - strict ISO date prefix (`YYYY-MM-DD`)
  - non-empty context text after the date
- Standardize format as `YYYY-MM-DD - <context>`, for example: `2026-04-15 - Beta release announced to pilot customers`.
- Render markdown in this order:
  - `## Summary`
  - `## Key Points`
  - `## Timeline`
  - `## References`
- Update canonical merge parsing/rendering to preserve and deduplicate timeline bullets across merges.
- Keep backward compatibility for existing files without `## Timeline` (treat as empty until next write).

## Scope

In scope:

- Update `DigestItem` types and JSON schema requirements.
- Update digest markdown renderer to include `## Timeline`.
- Update canonical topic parser/writer to read and write `## Timeline`.
- Enforce strict ISO date + non-empty context validation for timeline entries in parsing/normalization.
- Update tests for schema parsing, markdown rendering, merge behavior, and legacy compatibility.
- Update README to document the new section and expected timeline format.

Out of scope:

- Timezone-aware datetime formats (for example `YYYY-MM-DDTHH:mm:ssZ`).
- Replacing the `references` structure or reference validation rules.
- Automatic extraction of dates from free-form summary/key point text.

## Implementation Notes

- Use a single validation rule for timeline entries so behavior is consistent in digest parsing and merge parsing.
- Recommended validation pattern: `^\d{4}-\d{2}-\d{2}\s+-\s+.+$`.
- Reject date-only entries (for example `2026-04-15`) because context is required.
- Maintain existing fallback style (`- None`) when timeline has no entries.
- Include `timeline` in digest identity hashing to avoid false no-op merges when timeline content changes.
- Update prompt template guidance (`prompts/digest.md`) so model output includes timeline entries in English with strict ISO dates and context details.

## Acceptance Criteria

- [ ] Digest structured output accepts missing `timeline` and defaults to an empty timeline.
- [ ] Timeline entries, when provided, are validated to match `YYYY-MM-DD - <context>` with non-empty context.
- [ ] Markdown output always includes `## Timeline` between `## Key Points` and `## References`.
- [ ] Canonical topic files preserve timeline entries across merges and remain idempotent for duplicates.
- [ ] Existing legacy files without timeline still parse and can be merged without errors.
- [ ] Tests cover valid/invalid timeline formats, render output, and merge behavior.
- [ ] README documents `## Timeline` and the required `YYYY-MM-DD - <context>` format.

## Risks / Considerations

- AI output may still occasionally produce non-ISO date text; validation and retry/error messaging must be clear.
- Strict format improves consistency but may reject ambiguous user-provided dates unless normalized upstream.
