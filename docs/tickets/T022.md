# T022 - Merge lineage metadata and slug-length controls

## Problem / Goal

- Topic files currently store `digested_note_paths` (formerly `merged_digest_ids`) using reference-derived IDs, which does not represent the actual merged digest notes.
- Some generated topic slugs are too long and create unwieldy file names.
- Digest notes only persist `merged: true|false`, so there is no direct link from a merged digest note to topic file(s) it affected.

## Proposed Change

- Redefine `digested_note_paths` to store merged digest-note identities (not references).
- Persist digest-note-to-topic linkage in digest note front matter via `merged_topic_paths`.
- Add a hard slug-length cap of 100 characters for topic file names.
- Update merge prompt guidance so `create_new.shortDescription` is concise and suitable for a slug capped at 100 chars.
- Preserve merge idempotency without using `digested_note_paths` for reference dedupe.

## Scope

In scope:

- Metadata contract changes for topic files:
  - `digested_note_paths` becomes a YAML list of merged digest-note IDs.
  - Digest-note ID format is project-relative note path (example: `notes/planning_2026-03-09_1.md`).
- Metadata contract changes for digest note files:
  - Add `merged_topic_paths` as a YAML array of merged topic file paths.
  - Keep `merged` boolean behavior unchanged.
- Merge pipeline wiring:
  - Capture digest note identity in merge flow.
  - Pass merged digest ID into topic merge composition.
  - Persist merged topic paths when marking a stage note as merged.
- Slug-length control:
  - Enforce 100-char max length in code-level slug generation.
  - Ensure final slug remains valid kebab-case and non-empty after truncation.
- Prompt/schema updates:
  - Update merge routing prompt text to request concise slug-friendly `shortDescription` up to 100 chars.
  - Optional parsing guard for overly long `shortDescription` (validate or normalize consistently).
- Tests and docs updates:
  - Update/expand tests for new metadata semantics and slug cap.
  - Update README section describing topic/stage metadata fields and meanings.

Out of scope:

- Migrating existing historical topic files/stage notes in-place.
- Introducing globally unique IDs for stage notes beyond existing deterministic file identity.
- Changing canonical `Summary/Key Points/Timeline/References` section structure.

## Non-goals

- Using references as persistent merge identity in topic metadata.
- Adding a reverse index database for lineage tracking.
- Changing merge confirmation UX.

## Implementation Notes

- `digested_note_paths` should always serialize as one YAML list item per merged digest-note ID; never as a single pipe-joined payload.
- Reference-based no-op detection should remain internal behavior (for idempotency) and no longer define front matter lineage fields.
- `merged_topic_paths` must support multiple values because one digest note can merge into multiple topic files.
- Paths should be stored project-relative for portability across environments.
- Keep backward compatibility when reading old stage/topic metadata:
  - Missing new fields should default safely.
  - Legacy `merged_digest_ids` values (reference-style) should not crash parsing and should map to `digested_note_paths` on write.

## Acceptance Criteria

- [ ] New merges append digest-note IDs (project-relative note paths) to topic `digested_note_paths`.
- [ ] Topic `digested_note_paths` serialize as YAML list entries, one per merged note ID.
- [ ] Stage-1 notes include `merged_topic_paths` after successful merge, containing all merged topic target paths.
- [ ] Existing `merged` behavior is preserved (`merge` still processes only unmerged digest notes).
- [ ] Topic slug/file name generation enforces max length of 100 characters.
- [ ] Merge routing prompt guidance is updated to ask for slug-friendly short descriptions within the 100-char limit.
- [ ] Re-merging equivalent content remains a no-op via internal dedupe logic (without relying on `digested_note_paths` reference entries).
- [ ] Tests are updated/added for metadata semantics, stage/topic linkage, and slug-length enforcement.
- [ ] README metadata documentation reflects the new lineage fields and semantics.
