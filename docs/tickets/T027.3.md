# T027.3 - Generic MCP runtime and import persistence

## Problem / Goal

- CLI scaffolding alone is insufficient without an MCP runtime and output persistence path.
- We need a robust, generic execution layer for action listing and tool calls, then convert results into digest-ready markdown files.

## Proposed Change

- Add MCP client module for local stdio JSON-RPC sessions.
- Support operations needed for import:
  - initialize session
  - list actions/tools
  - call selected tool with JSON args
- Implement output normalization to markdown and write import files.
- Expose storage helpers through `src/files.ts` for CLI orchestration.

## Scope

In scope:

- Process spawn lifecycle and request timeout handling.
- JSON-RPC response/error handling and clean shutdown.
- Default output path generation under `<project>/imports`.
- Markdown template for imported content metadata and payload.

Out of scope:

- Source-specific post-processing for Slack/Confluence semantics.
- Caching or incremental sync state.

## Implementation Notes

- Keep rendering generic and deterministic for any MCP tool result shape.
- Prefer safe fallback rendering when result payload is partially structured.
- Ensure no file writes occur in dry-run mode.

## Acceptance Criteria

- [ ] Import action listing successfully retrieves actions from local MCP server.
- [ ] Import tool execution successfully calls MCP tool with parsed JSON args.
- [ ] Tool output is persisted to markdown with stable metadata fields.
- [ ] Default output path under `<project>/imports` is deterministic; explicit `--output` is honored.
- [ ] Runtime failures (timeout, JSON-RPC error, spawn failure) surface actionable errors.
