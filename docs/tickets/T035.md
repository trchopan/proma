# T035 - Do not promote every merged PR into a decision topic

## Problem / Goal

- Proma currently auto-promotes many git-derived digest notes into `decision` topics, including routine implementation changes.
- This creates low-value decision artifacts (for example icon tweaks, version bumps, small config edits) that dilute durable project knowledge.
- Goal: introduce a promotion threshold so only meaningful, reference-worthy git changes become new decision topics.

## Proposed Change

- Add a deterministic routing guard in merge flow for `decision` + `git` items when creating a new topic.
- Apply the agreed policy as `new only`: block low-signal routine changes from creating new `topics/decision/*.md` files, while preserving updates to existing decision topics.
- Use signal-based heuristics for the guard:
  - small/routine change cues in summary/key points,
  - missing rationale/tradeoff cues,
  - absence of architecture/infrastructure/policy/significant-impact cues.
- When guard conditions match, treat merge as no-op for topic creation (`hasChanges: false`) and keep digest note handling consistent with existing merge flow semantics.

## Scope

In scope:

- Guard logic in topic merge path for routine git decision-note promotion.
- Tests that verify both blocked and allowed promotion cases.
- README behavior note documenting the decision-promotion threshold.

Out of scope:

- Reclassifying digest category from `decision` to `planning`/`research`.
- Introducing new CLI flags to tune threshold behavior.
- Refactoring AI prompt contracts or adding network-dependent decision scoring.
- Retrospective cleanup of already-created trivial decision topics.

## Non-goals

- Do not change dry-run guarantees.
- Do not change markdown section contracts for digest/topic/report files.
- Do not block updates to existing decision topics in this ticket.

## Implementation Notes

- Keep guard deterministic and local (string/signal heuristics only).
- Enforce guard only when all conditions hold: category is `decision`, source is `git`, and merge target is effectively new.
- Preserve existing behavior for non-git sources and for decision notes that include rationale or significant impact signals.
- Add regression tests for false positives/false negatives around rationale and impact keywords.

## Acceptance Criteria

- [x] Routine git decision digest notes do not create new `topics/decision/*.md` files when threshold conditions are not met.
- [x] Git decision notes with explicit rationale/tradeoff signals are still eligible to create new decision topics.
- [x] Git decision notes with architecture/infrastructure/policy or clearly significant impact signals are still eligible to create new decision topics.
- [x] Existing `update_existing` decision-topic merges remain unaffected by this threshold.
- [x] Non-git decision sources remain unaffected by this threshold.
- [x] Tests are updated in `tests/topic-merge-service.test.ts` and `tests/files.test.ts`.
- [x] README documents decision-promotion threshold behavior for merge.
