# T032 - Improve topic consolidation with source-agnostic workstream routing

## Problem / Goal

- Planning topics are fragmented into too many narrow files (often one update per note/PR), making review and longitudinal tracking harder.
- Current merge flow relies heavily on model routing choice after top-N ranking, and lacks a deterministic anti-fragmentation guard.
- Goal: make topic structure more reviewable by consolidating into stable workstream topics across all sources, while preventing incorrect cross-cycle merges.

## Evidence

- `tmp/sample-project/topics/planning/` contains multiple near-duplicate workstream topics (for example several dependency-stream updates under one release window).
- Existing routing prompt does not explicitly enforce anti-fragmentation/workstream-level consolidation.
- Current flow has no deterministic post-router override when AI selects `create_new` despite a strong existing match.
- Post-implementation review found cross-project contamination where one project coordination topic absorbed another project's release notes due to insufficient identity splitting.

## Proposed Change

- Introduce source-agnostic topic cohesion logic using domain/workstream/timebox signals from digest content.
- Treat timebox as a hard split key (for example release/sprint/week): never merge across timebox boundaries when detected.
- Strengthen routing prompt guidance to prefer updating existing workstream topics over creating note-specific topics.
- Enrich candidate context passed to routing with stability signals (for example merged-note count, recency) to favor established canonical topics.
- Add deterministic post-router consolidation guard:
  - if AI chooses `create_new` but top candidate is a high-confidence same-timebox workstream match, coerce to `update_existing`.
- Improve ranking weights to emphasize durable workstream semantics and de-emphasize volatile identifiers (PR/ticket numbers).
- Add project/product identity anchors as a hard split key to prevent cross-project merges (for example `project-atlas-api` vs `project-orion-web`).

## Scope

In scope:

- Prompt guidance updates for merge routing behavior.
- Candidate model/context enrichment for routing decisions.
- Deterministic anti-fragmentation guard in merge orchestration.
- Ranking/scoring adjustments for source-agnostic cohesion.
- Tests covering consolidation and safe split behavior.
- README updates documenting new merge/topic structuring policy.

Out of scope:

- Bulk migration or auto-rewrite of all historical topic files.
- Manual curation of existing topic names/slug conventions beyond what merge flow produces.
- Changing digest category taxonomy.

## Non-goals

- Do not remove AI routing; deterministic logic is a safety guard, not a full replacement.
- Do not merge semantically distinct workstreams just to reduce file count.
- Do not merge across timeboxes when a clear boundary exists.

## Implementation Notes

- Keep behavior deterministic and testable.
- Preserve current CLI/dry-run contracts.
- Ensure tie-breakers remain stable (slug lexical fallback) to avoid nondeterministic merges.
- Prefer minimal API surface changes unless required by candidate context enrichment.

- [x] Timebox is treated as a hard split key; notes from different timeboxes do not merge into the same topic.
