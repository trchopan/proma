# T024.1 - Project config module loading for digest source settings

## Problem / Goal

- The CLI has no project-level config loading path for digest source policy.
- We need deterministic module discovery and parsing so later plugin-related config can build on the same foundation.

## Proposed Change

- Add project config loader that searches project root for:
  - `proma.config.ts`
  - `proma.config.mjs`
  - `proma.config.js`
- Enforce precedence order: `.ts` first, then `.mjs`, then `.js`.
- Support default-exported object config with `digest.allowedSources`.
- Add resolver function that returns effective allowed sources as union(defaults + custom).

## Scope

In scope:

- New config types and loader utilities.
- Validation/normalization for configured `digest.allowedSources`.
- CLI wiring to resolve config from the command's `--project` root.

Out of scope:

- Plugin execution hooks.
- Dynamic config reload/watch behavior.

## Implementation Notes

- Normalize configured source values with trim + lowercase.
- Ignore empty configured entries.
- Keep resolved list deterministic for test stability.

## Acceptance Criteria

- [ ] Config module resolution follows `ts > mjs > js` precedence.
- [ ] Missing config returns defaults only.
- [ ] Valid config with custom sources extends defaults.
- [ ] Invalid config shape yields clear CLI/runtime error.
