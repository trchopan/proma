# T017.1 - Build plugin guide foundation and architecture narrative

## Problem / Goal

- A recipes-only document is not enough for maintainability; users need a mental model before they patch prompts.
- The guide should clearly explain plugin lifecycle from config discovery to setup execution and registry validation.

## Proposed Change

- Draft the first version of `PLUGIN_SYSTEM.md` with a structured outline and architecture sections.
- Document key concepts: operations, registry, plugin API, setup order, and fallback behavior when no config file exists.

## Scope

In scope:

- Intro section and intended use-cases.
- Config file discovery section (`proma.config.ts`, `proma.config.js`, `proma.config.mjs`).
- API reference stubs with signatures and semantics.
- Execution flow diagram or step list (text-based is fine).

Out of scope:

- Full set of example recipes (handled in T017.2).

## Implementation Notes

- Keep terminology identical to source types (`PromptPlugin`, `PromptPluginApi`, `PromptRegistry`).
- Explicitly mention first-match config loading behavior and declaration-order plugin execution.
- Use concise, direct language suitable for external users.

## Acceptance Criteria

- [ ] `PLUGIN_SYSTEM.md` exists with clear section structure.
- [ ] Guide explains architecture and lifecycle without contradicting source code.
- [ ] Config discovery and ordering rules are documented accurately.
- [ ] API reference includes `getRegistry`, `patchOperation`, and `overrideOperation`.
