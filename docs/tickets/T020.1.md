# T020.1 - Section config model and validation

## Problem / Goal

- The system lacks a structured high-level config for sections, so users are forced into brittle low-level schema mutations.

## Proposed Change

- Introduce shared section definition types and config shape for `digest`, `topic`, and `report`.
- Validate section definitions during config load.
- Add utility to derive strict JSON Schema objects from section definitions.

## Scope

In scope:

- Type definitions for section model (`string` and `string[]`).
- Validation rules (required fields, key uniqueness, key format, allowed types, non-empty descriptions).
- Schema builder utilities and tests.

Out of scope:

- Wiring section model into digest/report/topic runtime behavior.
- Markdown parser/renderer changes.

## Non-goals

- Supporting nested object sections or custom JSON types.

## Implementation Notes

- Keep validation errors actionable and tied to config file context.
- Preserve compatibility with existing config files that only use `plugins`.

## Acceptance Criteria

- [ ] `proma.config.*` supports a validated `sections` property with `digest`, `topic`, and `report` arrays.
- [ ] Invalid section config fails fast with descriptive errors.
- [ ] Schema utility produces strict object schema from section definitions.
- [ ] Tests cover valid and invalid config scenarios.
