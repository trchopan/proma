# T025 - Category-specific topic structures end-to-end

## Problem / Goal

- Topic files currently use one shared body shape (`Summary/Key Points/Timeline/References`) across all categories.
- This loses category semantics and causes awkward content placement for `discussion`, `research`, and `planning`.
- Goal: fully separate topic structures by category and propagate that separation through AI merge contracts, runtime merge behavior, storage parsing, and report input extraction.

## Proposed Change

- Define category-specific topic section contracts:
  - `discussion`: `Summary`, `Context/Background`, `Resolution`, `Participants`, `References`
  - `research`: `Summary`, `Problem Statement`, `Research Plan`, `Key Findings`, `Person in Charge`, `References`
  - `planning`: `Summary`, `Objectives / Success Criteria`, `Scope`, `Deliverables`, `Plan`, `Timeline`, `Teams/Individuals Involved`, `References`
- Replace generic merge-content output contract with category-discriminated contracts.
- Update prompt/schema/parser stack so `merge_content` is category-aware and strict per category.
- Update merge persistence and topic/report readers to use category-specific fields.
- Keep digest-note format unchanged (`Summary/Key Points/Timeline/References`) to avoid changing stage-1 note contract.

## Scope

In scope:

- Category-aware topic body parser/renderer and merge model.
- Category-aware merge-content schemas, prompt text, parser validation, and type contracts.
- CLI merge wiring and fallback behavior for category-specific content.
- Topic candidate parsing and report input parsing using category-specific sections.
- Tests and README updates for new structure.

Out of scope:

- Migration of historical topic markdown files.
- Changes to digest note markdown format.
- Changes to report output schema/format.

## Non-goals

- Auto-rewriting old topic files to the new section format.
- Introducing additional categories beyond `planning|research|discussion`.

## Implementation Notes

- Prefer dual-read compatibility for topic parsing during transition (read legacy and new headings), but always write new category-specific headings.
- Maintain deterministic fallback behavior when semantic merge generation fails or output is invalid.
- Ensure timeline validation remains strict where timeline exists (`planning` only in new structure).

## Acceptance Criteria

- [ ] Topic files are written with category-specific section structures exactly matching product requirements.
- [ ] `merge_content` schema/prompt/parser/types are category-specific and no longer rely on one shared keyPoints/timeline contract.
- [ ] Merge runtime persists and merges category-specific fields correctly.
- [ ] Topic candidate extraction and report context extraction parse category-specific sections correctly.
- [ ] Digest note processing remains backward compatible and unchanged in structure.
- [ ] Tests are updated/added for category-specific merge flows and parser validation.
- [ ] README documents category-specific topic section contracts.
