# T015.3 - Add plugin API and `proma.config.ts` loading

## Problem / Goal

- Users need an in-repo mechanism to customize prompts/schemas without modifying core source files.

## Proposed Change

- Define plugin interface and plugin API methods for registry patch/override.
- Implement config loading from `proma.config.ts`.
- Apply plugins in deterministic order and return the composed registry.

## Detailed Design

- Add `PromptPlugin` contract:
  - `name: string`
  - `setup(api): void | Promise<void>`
- Add `PromptPluginApi` contract methods:
  - `getRegistry()`
  - `patchOperation(kind, patcher)`
  - `overrideOperation(kind, next)`
- Add `loadPromptRegistry()` flow:
  1. create built-in registry
  2. discover and import `proma.config.ts` from workspace root
  3. read `plugins` array
  4. execute plugin setup in declaration order
  5. return composed registry for validation/execution
- Config format for v1:
  - `export default { plugins: PromptPlugin[] }`

## Scope

In scope:

- Plugin type contracts.
- Config discovery/loading.
- Plugin execution lifecycle.

Out of scope:

- Runtime sandboxing or remote plugin sources.

## Implementation Notes

- Keep v1 to operation mutation of existing `digest|merge|report` entries.
- Emit readable errors for invalid plugin exports or failed setup hooks.
- Include plugin name in all thrown errors when available.
- Keep plugin side effects explicit and local to registry composition.

## Risks / Edge Cases

- Missing config file should not fail startup; use built-ins by default.
- Invalid default export shape should fail with clear contract guidance.
- Async plugin setup errors should preserve stack details in verbose logs.

## Acceptance Criteria

- [ ] `proma.config.ts` can register multiple plugins.
- [ ] Plugins can patch/override existing operation definitions.
- [ ] Plugin execution order is stable and documented.
- [ ] Plugin load failures surface actionable startup errors.
- [ ] Missing `proma.config.ts` falls back to built-in registry without failure.
