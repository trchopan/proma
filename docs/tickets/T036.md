# T036 - Prevent near-duplicate planning topics in merge

## Problem / Goal

- Merge can create near-duplicate planning topics with almost identical titles/slugs when AI returns `create_new` for a digest item that is semantically adjacent to an existing topic.
- This produces fragmented canonical knowledge (similar filenames, overlapping labels, unrelated/partially related payloads) and makes report generation noisier.
- Goal: introduce deterministic guardrails so `create_new` is either consolidated to an existing topic when safe, or forced to produce a clearly differentiated new topic identity.

## Evidence

- `tmp/sample-portal/topics/planning/sample-portal-weekly-highlights-xlt-beta-rollout-sentry-migration-and-engineering-updates.md`
- `tmp/sample-portal/topics/planning/sample-portal-weekly-highlights-xlt-beta-rollout-sentry-migration-dependency-and-engineering-updates.md`
- Merge run log: `logs/2026-03-15/2026-03-15T06-51-32-384Z_merge_54962.jsonl`

## Proposed Change

- Add a deterministic near-duplicate guard in merge consolidation for `create_new` targets:
  - compare normalized topic identity against ranked/existing candidates,
  - detect high similarity in slug/topic tokens,
  - prevent creating a second topic when identity overlap exceeds threshold and no hard split requires separation.
- Preserve existing hard split boundaries (timebox and project/product anchors) while improving create_new behavior:
  - if hard split exists, allow `create_new` but enforce differentiators in topic identity.
- Add topic identity disambiguation policy for allowed `create_new` cases:
  - require at least one durable differentiator in `shortDescription/topic` (for example `release-1-13-0`, repo/service anchor, issue/PR anchor when stable enough for routing context).
- Tighten merge routing prompt/schema guidance so AI is explicitly instructed to avoid near-duplicate topic names and include distinguishing scope markers when creating new topics.

## Scope

In scope:

- Deterministic duplicate-similarity checks and routing fallback behavior in merge consolidation.
- Prompt-level guidance updates for topic naming and create_new disambiguation.
- Regression tests for duplicate prevention and preserved hard-split behavior.
- README behavior notes for merge topic identity rules.

Out of scope:

- Bulk migration/renaming of historical duplicate topic files.
- Introducing new CLI flags for duplicate thresholds.
- Rewriting ranking/scoring architecture beyond targeted guardrails.

## Non-goals

- Do not change digest markdown contract.
- Do not weaken hard split guarantees for timebox/project anchors.
- Do not add nondeterministic post-processing.

## Implementation Notes

- Build a normalized identity feature set from both AI target and existing candidates:
  - normalized slug tokens,
  - normalized topic tokens,
  - key differentiator tokens (timebox, anchor, release markers).
- Add helper logic in `chooseConsolidatedTarget` path:
  - for `aiTarget.action === create_new`, detect near-duplicate candidate identity,
  - if near-duplicate + no hard conflict => convert to `update_existing` for the best candidate,
  - if near-duplicate + hard conflict => keep `create_new` but require differentiated identity (fallback normalization strategy if AI target lacks one).
- Keep behavior deterministic and testable; no network or additional AI calls.

## Acceptance Criteria

- [ ] Merge does not create a new planning topic when `create_new` name/slug is near-duplicate of an existing candidate and no hard split boundary applies.
- [ ] Merge still creates new planning topics when hard split boundaries require separation (timebox/project anchors), but with clear differentiated identity.
- [ ] Existing strong-match promotion logic to `update_existing` remains intact.
- [ ] Regression tests cover near-duplicate suppression, hard-split preserve-create_new, and disambiguated create_new naming behavior.
- [ ] CLI merge tests verify the final target action/slug chosen under duplicate-like inputs.
- [ ] README documents near-duplicate guardrail behavior for merge topic creation.
- [ ] `bun run check` passes after implementation.
