# T031 - Fix empty planning participants and enforce identity format

## Problem / Goal

- Planning topic files can end up with `## Teams/Individuals Involved` as `- None` even when source content explicitly includes ownership (for example merged-by actor from Git PR metadata).
- Example observed: `tmp/media-portal/topics/planning/media-portal-fe-release-1-12-0-delete-modal-text-ul-update.md` has no participants, while imported PR data for `#2590` includes `merged_by.login = long-tran`.
- Goal: preserve ownership signals through digest + merge and render canonical participant entries in planning topics.

## Evidence

- `tmp/media-portal/topics/planning/media-portal-fe-release-1-12-0-delete-modal-text-ul-update.md`
- `tmp/media-portal/notes/planning_2026-03-14_16.md`
- `tmp/media-portal/imports/2026-03-14T02-46-52-746Z_github_pr_get_pr-2590.md`

## Proposed Change

- Strengthen digest prompt guidance to retain explicit ownership/actor cues from source content in digest output.
- Strengthen planning merge-content guidance so `teamsIndividualsInvolved` is populated whenever named people/teams exist in incoming content.
- Add deterministic fallback extraction in merge logic for planning category from incoming summary/key points (for example `merged by <handle>`, `author: <handle>`, `assignee: <handle>`, `@<handle>`).
- Enforce canonical participant output format:
  - Full info: `Display Name (platform:identity handle)`
  - Missing display name: `(platform:identity handle)`
- Use source label directly for platform namespace (for example `git`, `slack`).

## Scope

In scope:

- Prompt guidance updates for digest and planning merge-content operations.
- Deterministic participant extraction/normalization in planning merge fallback.
- Regression tests for prompt wording and merge participant extraction formatting.
- README contract update for planning participant format behavior.

Out of scope:

- Backfilling all existing historical topic files automatically.
- Adding new frontmatter fields for participant metadata provenance.
- Renaming source label `git` to `github`.

## Non-goals

- Do not alter timeline format contracts.
- Do not change allowed source taxonomy.
- Do not weaken strict structured-output (`json_schema`) requirements.

## Implementation Notes

- Keep merge behavior deterministic and idempotent.
- Prefer canonical de-duplication so repeated merges do not create participant duplicates.
- Parser should avoid extracting false positives from generic prose when no handle-like identity is present.

## Acceptance Criteria

- [x] Planning merges include participant entries when incoming digest text contains explicit ownership cues.
- [x] Participant rendering follows exact canonical forms:
  - [x] `Display Name (platform:identity handle)`
  - [x] `(platform:identity handle)`
- [x] For `source: git`, participant namespace is `git` (not `github`).
- [x] Prompt tests assert ownership-preservation instructions are present.
- [x] Merge service tests cover deterministic extraction and formatting for at least `git` and `slack` examples.
- [x] `bun run check` passes after implementation.
