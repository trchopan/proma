# T020.2 - Section-driven digest/report pipeline

## Problem / Goal

- Digest and report are currently hardcoded to fixed fields and section names, so custom section config cannot flow through prompt schema, parser, and markdown output.

## Proposed Change

- Build digest/report response schema from configured section definitions.
- Parse model output using section definitions.
- Render markdown sections dynamically using section definitions.

## Scope

In scope:

- Registry wiring for digest/report schema generation.
- Digest/report parser updates for `string` and `string[]` section values.
- Digest/report markdown renderer updates preserving stable section order.
- Regression and feature tests for default and custom section behavior.

Out of scope:

- Topic canonical parsing and merge behavior.
- Low-level plugin API removal.

## Non-goals

- Changing command UX or CLI flags.

## Implementation Notes

- Preserve current output exactly when `sections.digest` / `sections.report` are not configured.
- Ensure parser normalization remains deterministic (trim, dedupe for arrays where applicable).

## Acceptance Criteria

- [ ] Digest schema/parsing/rendering follow configured digest sections.
- [ ] Report schema/parsing/rendering follow configured report sections.
- [ ] Existing default digest/report output remains unchanged without custom config.
- [ ] Tests cover both default and custom section paths.
