# T015.4 - Add registry validation and startup guardrails

## Problem / Goal

- Plugins can introduce invalid operation shapes that otherwise fail later during model calls.

## Proposed Change

- Add a validator for the composed registry.
- Validate required operations exist (`digest`, `merge`, `report`) and each has:
  - prompt builder
  - parser
  - strict structured output response format compatibility
- Integrate validation into startup path before command execution.

## Detailed Design

- Add `validatePromptRegistry(registry)` with checks:
  - required keys are present
  - each operation has `kind` matching key
  - `buildPrompt` and `parseResponse` are functions
  - built prompt returns `messages` array and `responseFormat`
  - response format type is `json_schema` for all v1 operations
  - schema payload includes `name`, `strict`, `schema`
- Run validation once after plugin composition and before command dispatch.
- Throw domain-specific error messages that include:
  - operation key (`digest|merge|report`)
  - violated contract field
  - plugin source if known

## Scope

In scope:

- Validation functions and startup integration.
- Error message quality.

Out of scope:

- Deep semantic validation of user prompt content.

## Implementation Notes

- Validation errors should include plugin/operation context where available.
- Keep validator free of network/model checks; it should be deterministic and fast.

## Risks / Edge Cases

- Overly strict checks could block valid future extensions; keep v1 checks explicit and documented.
- Validation that executes `buildPrompt` must use safe minimal sample contexts and avoid side effects.

## Acceptance Criteria

- [ ] Invalid registries fail before any AI request is sent.
- [ ] Error messages identify which operation and contract failed.
- [ ] Valid built-in + plugin registries pass validation.
- [ ] Startup logs indicate when custom plugin-composed registry passes validation.
