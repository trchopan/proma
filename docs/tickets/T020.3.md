# T020.3 - Custom topic sections for merge flow

## Problem / Goal

- Topic files are currently canonicalized with fixed sections only, so user-defined `topic` sections cannot be parsed, rendered, or merged predictably.

## Proposed Change

- Make canonical topic section parsing/rendering driven by configured `sections.topic`.
- Apply deterministic merge behavior for custom sections:
  - `string`: keep existing non-empty value, fallback to incoming.
  - `string[]`: union + dedupe with stable ordering.

## Scope

In scope:

- Canonical topic parser and renderer refactor to section-driven model.
- Topic merge strategy updates for generic section values.
- Compatibility with existing topic files when custom config is absent.
- Tests for section-aware topic round-trip and merge behavior.

Out of scope:

- Changes to front matter metadata contract.
- Changes to topic routing output schema.

## Non-goals

- Semantic merge heuristics per custom field beyond generic rules.

## Implementation Notes

- Preserve existing identity/no-op merge safeguards.
- Maintain deterministic markdown output ordering from section config.

## Acceptance Criteria

- [ ] Topic canonical parsing supports configured sections.
- [ ] Topic canonical rendering supports configured sections.
- [ ] Topic merge applies generic deterministic strategy by section type.
- [ ] Existing topic behavior remains unchanged without custom config.
