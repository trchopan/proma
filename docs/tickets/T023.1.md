# T023.1 - Single-target merge routing contract

## Problem / Goal

- Current routing allows multiple targets per digest item, which increases duplicate/overlapping merges across topic files.
- We need one primary target to reduce contamination and make merge behavior predictable.

## Proposed Change

- Update merge routing schema from `targets: []` to a single `target` object.
- Update parser/generator functions and merge orchestration to consume exactly one target per digest note.
- Preserve existing confirmation UX and dry-run behavior.

## Scope

In scope:

- `TOPIC_ROUTING_RESPONSE_SCHEMA` update in `src/digest.ts`.
- Parser update (`parseTopicRoutingResponse`) to return one normalized target.
- Generation API update (`generateTopicTargets` -> single target result shape or equivalent).
- CLI merge flow update to process only one prepared plan per digest note.
- Tests updated for schema, parsing, and CLI merge behavior.

Out of scope:

- Candidate ranking and prompt context enrichment.
- Semantic merge content generation.
- Tag governance improvements.

## Implementation Notes

- Keep backward compatibility errors clear if model returns legacy multi-target shape.
- Ensure unknown slug validation for `update_existing` remains enforced.

## Acceptance Criteria

- [ ] Merge routing schema requires exactly one `target`.
- [ ] Parsing rejects empty or invalid target payloads with actionable errors.
- [ ] CLI performs at most one merge write per digest note.
- [ ] Existing merge confirmation and dry-run output remain functional.
- [ ] Tests pass for updated single-target contract.
