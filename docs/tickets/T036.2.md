# T036.2 - Create_new identity differentiation under hard split

## Problem / Goal

- Even when hard split boundaries correctly force `create_new`, topic names can still be too similar to existing topics.
- This reduces discoverability and makes intent ambiguous.
- Goal: ensure forced `create_new` topics carry clear identity differentiators.

## Proposed Change

- When hard split blocks consolidation and AI target resembles an existing topic identity:
  - keep `create_new`,
  - enforce differentiated naming by injecting/retaining stable scope markers in `shortDescription`/`topic` (timebox, anchor, release marker, or similarly durable identifier).
- Keep slug normalization contracts intact (`slugifyTopic`, length cap).

## Scope

In scope:

- Deterministic name/slug differentiation policy for hard-split create_new cases.
- Fallback behavior when AI target lacks distinguishing tokens.
- Tests for differentiated output in hard-split duplicate-like scenarios.

Out of scope:

- Prompt contract/schema changes.
- Historical topic renaming.

## Non-goals

- Do not weaken hard split semantics.
- Do not introduce random suffixes.

## Implementation Notes

- Prefer durable markers already present in digest/candidate signals (release, project/product anchor, key scope token).
- Ensure generated names remain human-readable and deterministic.
- Keep behavior constrained to `create_new` paths impacted by near-duplicate detection + hard split conflict.

## Acceptance Criteria

- [ ] Hard-split `create_new` routes keep action as `create_new`.
- [ ] New topic identity is clearly differentiated from conflicting near-duplicate candidate(s).
- [ ] Differentiation logic is deterministic and stable across reruns.
- [ ] Tests cover representative hard-split scenarios and verify resulting target identity.
