# T015.1 - Define prompt control-plane core types

## Problem / Goal

- There is no single typed contract connecting prompt construction, response schema format, and parser output per process flow.

## Proposed Change

- Introduce control-plane core types for:
  - processing kinds: `digest`, `merge`, `report`
  - operation definitions (prompt builder + response parser)
  - operation context maps
  - registry and runtime contracts

## Detailed Design

- Define `ProcessingKind = "digest" | "merge" | "report"`.
- Define `PromptContextMap` with explicit per-kind context payload:
  - `digest`: input text and multimodal image parts metadata
  - `merge`: digest item payload + candidate topic list payload
  - `report`: period + input/base context payloads
- Define `OperationDefinition<TKind, TOutput>`:
  - `kind`, `version`
  - `buildPrompt(context) -> { messages, responseFormat, temperature? }`
  - `parseResponse(raw) -> TOutput`
- Define `PromptRegistry` keyed by all three processing kinds.
- Define execution/runtime helper types for model/logger threading.

## Scope

In scope:

- Type definitions and minimal shared interfaces.

Out of scope:

- Concrete operation implementations and plugin loading.

## Implementation Notes

- Keep generics constrained to the three processing kinds.
- Ensure prompt build result maps directly to `createChatCompletion` input shape.
- Prefer narrow object shapes over permissive index signatures for plugin safety.
- Keep exported types in a single module to avoid circular imports in later subtickets.

## Risks / Edge Cases

- Generic type complexity can make plugin authoring hard if over-abstracted.
- Context shape drift between command handlers and operation definitions can cause runtime mismatch.

## Acceptance Criteria

## Acceptance Criteria

- [ ] A new shared type module defines operation and registry contracts.
- [ ] Contracts support all three process kinds without `any`-driven APIs.
- [ ] Types are consumable by built-in operations and plugin APIs.
- [ ] Each operation context is explicit and cannot be substituted with another kind's context at compile-time.
