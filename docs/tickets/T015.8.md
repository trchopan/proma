# T015.8 - Document registry-driven prompting and plugin setup

## Problem / Goal

- Users and maintainers need clear guidance for the new centralized prompting architecture and plugin customization model.

## Proposed Change

- Update docs to explain:
  - built-in operation registry model
  - `proma.config.ts` plugin setup
  - plugin execution order and override semantics
  - constraints and best practices for schema/parser customization

## Documentation Plan

- README updates:
  - architecture section for control-plane modules
  - configuration section for `proma.config.ts`
  - plugin author quick-start snippet
  - migration note from legacy markdown prompt files
- Add one practical example plugin that customizes report system prompt while preserving schema/parser.
- Add warning notes for risky customizations:
  - schema changes must stay aligned with parser behavior
  - parser overrides should throw readable errors
  - plugin order affects final behavior

## Scope

In scope:

- README architecture and usage sections.
- Minimal plugin authoring example.

Out of scope:

- Full plugin cookbook with advanced recipes.

## Implementation Notes

- Include one concise end-to-end example that customizes report prompting.
- Keep docs aligned with actual exported types and function names to avoid drift.

## Risks / Edge Cases

- Example config drift after refactors can break copy-paste setup for users.
- Missing mention of plugin ordering can lead to subtle override bugs.

## Acceptance Criteria

- [ ] README reflects registry + plugin architecture.
- [ ] A `proma.config.ts` example is included.
- [ ] Docs describe supported plugin customization points and constraints.
- [ ] Legacy markdown template references are removed.
- [ ] Docs include a migration section for users with existing prompt customizations.
