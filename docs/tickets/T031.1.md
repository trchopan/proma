# T031.1 - Harden participant normalization for malformed owner strings

## Problem / Goal

- Planning topic `Teams/Individuals Involved` can contain malformed and duplicate participant entries when upstream digests include Slack owner text patterns such as `<@U...>.tran` and handle variants with `@` in canonical payload.
- Observed output includes low-quality display names and duplicates, for example `(.tran) (slack:U05T1QE6CJ0)` and both `ivy.example (slack:@ivy.example)` + `(slack:ivy.example)` in the same topic.
- Goal: keep participant rendering canonical, deterministic, and deduplicated even when incoming AI/upstream strings are noisy.

## Evidence

- `tmp/sample-portal/topics/planning/sample-portal-workstream-status-and-coordination.md`
- `tmp/sample-portal/notes/planning_2026-03-15_4.md`
- `tmp/sample-portal/notes/planning_2026-03-15_5.md`

## Proposed Change

- Accept optional leading `@` in canonical identity parsing for both forms:
  - `Display Name (platform:@handle)`
  - `(platform:@handle)`
- Normalize parsed handles via existing `normalizeIdentityHandle` to ensure canonical output stays `platform:handle` without `@`.
- Sanitize display names before rendering canonical identities:
  - trim and unwrap a single outer parenthesis pair,
  - reject malformed display names that are punctuation-like owner suffix artifacts (for example `.tran`, `.yun`),
  - fallback to handle-only canonical form when display name is invalid.
- Preserve deterministic identity-keyed dedupe behavior so handle-equivalent forms collapse to one participant entry.

## Scope

In scope:

- Participant canonical parser hardening in planning merge normalization.
- Display-name sanitation before canonical identity formatting.
- Regression tests for optional-`@` canonical parsing and malformed display-name cleanup.

Out of scope:

- Changes to import markdown renderer or MCP/GitHub payload format.
- Backfill/rewrite of historical topic files already generated.
- New participant metadata fields in frontmatter.

## Non-goals

- Do not change source taxonomy (`git`, `slack`, etc.).
- Do not relax strict structured-output contract (`json_schema`).

## Implementation Notes

- Keep behavior deterministic and idempotent across repeated merges.
- Keep canonical participant contract from T031 unchanged:
  - `Display Name (platform:identity handle)`
  - `(platform:identity handle)`
- Use merge fallback extraction + normalization as final quality gate for noisy upstream participant text.

## Acceptance Criteria

- [x] Canonical parser accepts `platform:@handle` and normalizes to `platform:handle`.
- [x] Duplicate identities differing only by `@` prefix collapse to one entry.
- [x] Malformed display-name artifacts from owner suffix patterns do not render as display names.
- [x] `tests/topic-merge-service.test.ts` includes regression coverage for the above cases.
