# T012 - Remove legacy compatibility behavior (hard break)

## Problem / Goal

- The project is new and should not carry compatibility behavior for legacy file shapes that do not exist in production use.
- Current code and tests still support legacy staged notes (missing front matter) and legacy topic append format (`## Digest Entries`).
- We want a hard-break posture: only canonical formats are accepted and maintained.

## Proposed Change

- Remove compatibility parsing/fallback logic for staged note metadata in `src/storage/stage-notes.ts`.
- Remove legacy append-format handling in `src/services/topic-merge.ts`.
- Delete legacy-focused tests and fixtures under `tests/`.
- Update documentation to reflect strict canonical-only behavior and no backward-compat commitments for this area.

## Scope

In scope:

- Require explicit front matter fields for staged notes (`category`, `source`, `merged`).
- Reject non-canonical staged note content instead of inferring fields from filename/body.
- Remove `## Digest Entries` migration path; merge logic operates on canonical sections only.
- Remove tests/fixtures that validate legacy compatibility behavior.
- Update active docs/tickets and README language to match hard-break policy.

Out of scope:

- Data migration scripts for hypothetical old repositories.
- Changes to unrelated compatibility wording outside this feature area if kept as historical record.

## Implementation Notes

- Keep parser failures explicit and actionable when metadata is missing/invalid.
- Preserve current canonical section contract: `## Summary`, `## Key Points`, `## Timeline`, `## References`.
- Ensure idempotency/no-op merge behavior still works for canonical inputs.

## Acceptance Criteria

- [ ] `listPendingStageOneDigestItems` no longer accepts staged notes without valid front matter metadata.
- [ ] `buildTopicMergeContent` no longer contains legacy append-format (`## Digest Entries`) compatibility behavior.
- [ ] Legacy compatibility tests/fixtures are removed and test suite remains green with `bun test`.
- [ ] Documentation reflects canonical-only behavior and hard-break intent for this project phase.

## Risks / Considerations

- Existing local scratch files created with old test fixtures may fail parsing after this change.
- Hard-break behavior must be communicated clearly in docs to prevent confusion during onboarding.
