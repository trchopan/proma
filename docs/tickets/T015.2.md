# T015.2 - Implement built-in operation registry

## Problem / Goal

- Built-in behavior for prompt/schema/parser is currently scattered, making centralized control impossible.

## Proposed Change

- Implement a built-in registry factory that defines operation entries for:
  - `digest` generation
  - `merge` target routing
  - `report` generation
- Each entry includes message building, strict `json_schema` response format, and parser mapping.

## Detailed Design

- `createBuiltInRegistry()` returns fully typed entries for three operation keys.
- `digest` entry:
  - Build system/user messages equivalent to current defaults.
  - Support multimodal user message composition (`text` + `image_url` parts).
  - Use `DIGEST_RESPONSE_SCHEMA` and `parseDigestItemsResponse`.
- `merge` entry:
  - Build prompt with serialized digest item and candidate topic text.
  - Use `TOPIC_ROUTING_RESPONSE_SCHEMA`.
  - Parse via topic routing parser with candidate-aware validation context.
- `report` entry:
  - Build prompt with period + input/base JSON payloads.
  - Use `REPORT_RESPONSE_SCHEMA` and `parseReportResponse`.

## Scope

In scope:

- Built-in operation definitions only.

Out of scope:

- Plugin mutation and CLI integration.

## Implementation Notes

- Reuse existing schema constants and parse functions from digest/report modules where practical.
- Preserve multimodal prompt support for digest images.
- Keep operation schema names stable (`digest_items`, `topic_routing_targets`, `project_report`) to minimize downstream diff.
- Keep default temperature unchanged unless explicitly overridden by later plugin/runtime config.

## Risks / Edge Cases

- Merge parser currently depends on candidate slug validation; registry context must preserve candidate list.
- Prompt text drift can unintentionally change quality; preserve intent-level parity with current prompts.

## Acceptance Criteria

- [ ] Built-in registry exposes complete definitions for all three operations.
- [ ] Response format remains strict schema-based structured output.
- [ ] Behavior parity is maintained for existing default prompting intent.
- [ ] Digest multimodal requests continue to include image content parts in user message payload.
