# T025.1 - Category-specific topic markdown contracts

## Problem / Goal

- Topic markdown utilities currently assume one generic section set and cannot model category-specific semantics.
- We need parsing/rendering contracts that support distinct fields for `discussion`, `research`, and `planning` while keeping deterministic output.

## Proposed Change

- Add category-aware topic data types and section maps in `src/markdown/canonical-topic.ts`.
- Implement category-aware extraction and build helpers for topic files.
- Keep existing digest-note parser helper intact for stage-note compatibility.
- Support reading both new and legacy topic headings where feasible, but emit only new category-specific headings on write.

## Scope

In scope:

- New category-specific data models and helper APIs.
- Deterministic markdown rendering order and fallback placeholders per section.
- Basic integration updates in callers that currently parse/build topic bodies.

Out of scope:

- AI merge-content schema/prompt/parser changes (covered in T025.2).
- Full merge orchestration behavior updates (covered in T025.3).

## Implementation Notes

- Preserve strict timeline entry parsing for planning timeline sections.
- Ensure references continue using `- <source>: <link>` lines.
- Avoid ambiguous heading matching; use exact heading names plus explicit legacy aliases.

## Acceptance Criteria

- [ ] Category-aware topic parser returns correct structured fields for each category.
- [ ] Category-aware topic renderer outputs required sections in exact order per category.
- [ ] Legacy heading variants can be parsed for topic inputs without crashing.
- [ ] Digest-note parser behavior remains unchanged.
