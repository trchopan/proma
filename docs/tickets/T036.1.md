# T036.1 - Near-duplicate detection and consolidation fallback

## Problem / Goal

- `create_new` routing can produce topic names/slugs that are near-duplicates of existing planning topics.
- This creates parallel canonical files with overlapping identities and fragmented context.
- Goal: add deterministic duplicate detection and auto-consolidate to `update_existing` when no hard split requires a new topic.

## Proposed Change

- Extend `chooseConsolidatedTarget` duplicate handling for `aiTarget.action === create_new`:
  - compute normalized similarity between AI target identity and candidate identities,
  - identify best near-duplicate candidate,
  - convert to `update_existing` when similarity threshold is met and hard split checks do not block reuse.
- Keep current timebox/anchor hard conflict behavior unchanged.

## Scope

In scope:

- Deterministic identity normalization and similarity scoring for topic/title slug text.
- Consolidation fallback (`create_new` -> `update_existing`) for near-duplicates without hard conflict.
- Unit/integration tests for duplicate suppression behavior.

Out of scope:

- AI prompt/schema updates.
- Forced renaming of create_new outputs when hard conflict exists.

## Non-goals

- Do not change merge content generation.
- Do not add new CLI flags.

## Implementation Notes

- Use token overlap/Jaccard-style similarity on normalized slug/topic text.
- Ignore volatile tokens where appropriate (numbers/PR-like markers) while preserving release/timebox signals.
- Reuse existing hard-conflict checks (timebox/anchors) before converting action.

## Acceptance Criteria

- [ ] Near-duplicate `create_new` targets are converted to `update_existing` when no hard split boundary applies.
- [ ] Hard split boundaries still prevent unsafe consolidation.
- [ ] Tests in `tests/files.test.ts` cover duplicate suppression and no-regression cases.
- [ ] CLI-level test in `tests/cli.test.ts` verifies final target action under duplicate-like input.
