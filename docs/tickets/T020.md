# T020 - Section-based plugin customization API

## Problem / Goal

- Current plugin customization is too low-level because users must manually mutate deeply nested Structured Output JSON Schema and keep parser/renderer behavior aligned.
- Provide a simple, constrained API so users can declare sections for `digest`, `topic`, and `report` using only key, type (`string` or `string[]`), and description.

## Proposed Change

- Add a first-class config surface for section definitions in `proma.config.*`.
- Generate schema + parse + markdown rendering behavior from section definitions, with safe defaults when no custom sections are provided.
- Keep existing advanced plugin hooks (`patchOperation` and `overrideOperation`) for power users.

## Scope

In scope:

- New section config model and validation.
- Prompt schema generation based on sections for digest/report.
- Section-aware parsing/rendering for digest, topic canonical markdown, and report.
- Generic topic merge behavior for custom section fields.
- Documentation updates and migration examples.

Out of scope:

- Arbitrary JSON Schema customization through the high-level API.
- New section types beyond `string` and `string[]`.
- Breaking/removing existing low-level plugin APIs.

## Non-goals

- Replacing all plugin capabilities with the section API.
- Introducing runtime UI for editing sections.

## Implementation Notes

- Default behavior must remain unchanged when `sections` config is omitted.
- Section order in config should define schema field order and markdown section order.
- Normalize section keys to markdown headings in a deterministic way.

## Acceptance Criteria

- [ ] Users can define digest/topic/report sections with `{ key, type, description }` in `proma.config.*` without editing raw JSON Schema.
- [ ] Digest and report prompt schemas are generated from section definitions.
- [ ] Digest/topic/report markdown parsing and rendering support custom sections.
- [ ] Topic merge logic supports custom sections with deterministic merge strategy.
- [ ] Existing behavior remains compatible when section config is not provided.
- [ ] Docs include clear quick-start examples and migration guidance.
