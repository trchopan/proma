# T005.3 - Pending note selection and merge orchestration

## Problem / Goal

- With split commands, merge execution must discover staged notes from disk and avoid reprocessing already merged notes.
- We need deterministic orchestration that updates merge state only when processing succeeds.

## Proposed Change

- Add staged-note listing under `<project>/notes/*.md` and filter to `merged != true`.
- For each pending note, run existing candidate discovery, AI routing, merge planning, confirmation, and write flow.
- Mark staged note `merged: true` after successful processing; keep `false` when merges are skipped or produce no applied writes.

## Scope

In scope:

- Pending note discovery and loading logic.
- Integration of pending notes into current topic merge loop.
- Persisting staged note merge state updates.

Out of scope:

- Parallel processing across notes.
- Retry queues or background scheduling.

## Implementation Notes

- Treat a note as complete only when all selected merge targets are either no-op or confirmed/applied without unresolved skips.
- Preserve user interaction model (diff preview per target).
- Keep logs explicit about skipped, no-op, and completed note outcomes.

## Acceptance Criteria

- [ ] `merge` processes only notes that are not yet merged.
- [ ] Already merged notes are skipped without AI calls.
- [ ] Completed notes are persisted with `merged: true`.
- [ ] Notes with rejected/skipped targets remain `merged: false`.
- [ ] Integration tests cover mixed pending/merged note sets.
