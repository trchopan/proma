# T015.7 - Add comprehensive tests for control-plane architecture

## Problem / Goal

- The new architecture needs confidence coverage across built-ins, plugin overrides, validation, and command integration.

## Proposed Change

- Add unit tests for:
  - registry construction
  - plugin patch/override composition
  - validation failures
  - runtime execution path
- Update integration tests to assert digest/merge/report flow behavior under the unified runtime.

## Test Plan

- Unit tests:
  - `createBuiltInRegistry()` returns all required operation keys.
  - plugin patch order is deterministic (later plugin wins where conflicts occur).
  - invalid plugin export shape fails with actionable error.
  - registry validator rejects missing keys and invalid response format contracts.
  - runtime executor threads model/logger/temperature correctly to chat client.
- Integration-level tests:
  - digest flow with text-only input
  - digest flow with image input
  - merge flow with candidate slug validation
  - report flow with period/input/base contexts
  - fallback path when config file is absent

## Scope

In scope:

- New and updated test cases for control-plane behavior.

Out of scope:

- End-to-end tests requiring external OpenAI network calls.

## Implementation Notes

- Use existing chat completion stubbing patterns for deterministic assertions.
- Keep test fixtures small and focused to avoid brittle prompt string matching.

## Risks / Edge Cases

- Plugin tests can become flaky if they depend on file import side effects; isolate config fixtures per test.
- Runtime tests should avoid over-snapshotting full prompts; assert key invariant substrings and schema shape.

## Acceptance Criteria

- [ ] Built-in registry behavior is covered by tests.
- [ ] Plugin overrides are covered by tests.
- [ ] Invalid plugin/registry configurations are covered by tests.
- [ ] Digest/merge/report command paths have passing tests using the new runtime.
- [ ] Test suite has no remaining reliance on removed prompt-template helper module.
