# T025.2 - Category-specific merge-content AI contract

## Problem / Goal

- The current `merge_content` contract is generic (`summary/keyPoints/timeline/references/tags`) and cannot enforce different required fields by category.
- We need strict, category-specific structured outputs to avoid semantic drift and mismatch with topic markdown structure.

## Proposed Change

- Convert merge-content input/output types to category-discriminated unions.
- Build category-specific JSON schemas for response validation.
- Update prompt building to include category-specific section guidance and constraints.
- Update merge-content response parser to validate by category and normalize outputs consistently.

## Scope

In scope:

- Type-level refactor for merge-content context/result.
- Schema and parser changes with strict validation.
- Prompt registry and prompt validation updates.
- Unit tests for valid/invalid category outputs.

Out of scope:

- Topic-file write/read implementation details (covered by T025.1/T025.3).
- CLI/storage side effects not directly tied to AI contract parsing.

## Implementation Notes

- Preserve allowed-source validation behavior for references.
- Keep timeline format constraint (`YYYY-MM-DD - <context>`) where timeline is part of category output.
- Ensure parsing errors are explicit and category-specific for debugging.

## Acceptance Criteria

- [ ] Merge-content type contracts are discriminated by category.
- [ ] JSON schema enforcement is category-specific and strict.
- [ ] Prompt instructions are category-specific and aligned with target markdown sections.
- [ ] Parser validates/normalizes per category and rejects invalid shapes.
- [ ] Existing digest/routing/report prompt operations remain unaffected.
