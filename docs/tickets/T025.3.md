# T025.3 - Runtime wiring for category-specific topic content

## Problem / Goal

- Even with category-aware markdown and AI contracts, runtime merge/storage paths must pass and persist those fields correctly.
- Current flow assumes shared `keyPoints/timeline` fields, so end-to-end category separation will be incomplete without orchestration updates.

## Proposed Change

- Update CLI merge path to handle category-specific merge-content outputs.
- Update topic merge service to merge/persist category-specific fields and render category-specific topic bodies.
- Update topic candidate parsing and report input extraction to consume category-specific topic structures.
- Preserve deterministic fallback and idempotency behavior.

## Scope

In scope:

- End-to-end merge payload handling from AI output to written topic markdown.
- Storage readers that support candidate ranking/report context from category-specific sections.
- Backward-compatible read behavior for existing topic files.

Out of scope:

- New ranking algorithm changes unrelated to field plumbing.
- Report format redesign.

## Implementation Notes

- Keep no-op detection semantics stable.
- If semantic merge fails, fallback content should still map cleanly into category-specific write model.
- Avoid changing digest note merge markers and lineage metadata in this ticket.

## Acceptance Criteria

- [ ] CLI merge passes category-specific merge-content outputs through to merge planning.
- [ ] Topic merge service writes category-specific markdown sections and merges fields correctly.
- [ ] Topic candidate reader and report input reader parse category-specific topic structures.
- [ ] Existing merge idempotency and fallback behavior remain intact.
