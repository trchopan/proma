# T033 - Replace `discussion` with `decision` topic taxonomy

## Problem / Goal

- The current taxonomy (`discussion`, `research`, `planning`) mixes process-oriented and artifact-oriented categories, which makes merge routing unstable and encourages catch-all discussion topics.
- Canonical topics should represent durable engineering knowledge artifacts. `discussion` captures transient conversation rather than durable outcomes.
- Goal: perform a hard taxonomy switch so canonical categories are exactly `research`, `decision`, and `planning`.

## Proposed Change

- Replace `discussion` with `decision` as a first-class digest/topic category across parsing, schema validation, prompting, merge content generation, markdown rendering, and report input scanning.
- Replace discussion canonical section model with decision-oriented sections and extraction/rendering logic.
- Update merge-content prompt/schema contracts to produce and validate decision fields.
- Update CLI/report category validation and fallback scan paths to `topics/planning`, `topics/research`, `topics/decision`.
- Apply a strict hard switch: no backward compatibility layer and no migration command in this ticket.

## Scope

In scope:

- Category enum/type replacement from `discussion` to `decision` in digest and merge contracts.
- Canonical topic parsing/rendering template replacement for decision category.
- Merge orchestration branch updates for merge-content input by category.
- Report input category validation and fallback directory scan updates.
- Tests updated for new taxonomy behavior and category validation.
- README updates documenting new taxonomy and section contracts.

Out of scope:

- Automatic migration of historical `topics/discussion/*` files.
- Legacy read compatibility for `discussion` category in parser/runtime behavior.
- New standalone migration CLI command.

## Non-goals

- Do not alter digest note markdown structure (`Summary`, `Key Points`, `Timeline`, `References`).
- Do not change dry-run guarantees or structured output requirements.
- Do not add silent fallback behavior that maps `discussion` to `decision` at runtime.

## Implementation Notes

- Keep structured output strict (`json_schema`) and update all related parser/schema/prompt surfaces together.
- Keep section names contract-level and deterministic in canonical topic rendering.
- Ensure all behavior-changing tests are updated in the same change set.
- User will manually clean up existing legacy `topics/discussion` data outside this ticket.

Execution order:

1. Complete `T033.1` to switch core category contracts (`discussion` -> `decision`) and structured-output schema/parser surfaces.
2. Complete `T033.2` to replace canonical topic section modeling and merge-category branching for decision artifacts.
3. Complete `T033.3` to update report scanning, docs, and regression tests, then run full validation gate.

## Acceptance Criteria

- [x] `DIGEST_CATEGORIES` supports `planning|research|decision` and no longer accepts `discussion`.
- [x] Digest and merge-content parsers reject `discussion` and accept `decision`.
- [x] Merge-content schema/prompt for `decision` is implemented and used by CLI merge orchestration.
- [x] Canonical topic extraction/rendering supports `decision` sections and removes discussion section logic.
- [x] Report fallback scan paths are exactly `topics/planning`, `topics/research`, and `topics/decision`.
- [x] CLI/report input category validation messages reflect `planning|research|decision`.
- [x] Tests in `tests/digest.test.ts`, `tests/files.test.ts`, and `tests/cli.test.ts` are updated for the new taxonomy.
- [x] `README.md` documents the new taxonomy and category-specific topic sections.
- [x] `bun run check` passes after implementation.
