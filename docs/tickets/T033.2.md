# T033.2 - Replace canonical discussion schema with decision schema in topic merge flow

## Problem / Goal

- Canonical topic section mapping and merge fallback logic still model `discussion` as a category, which conflicts with the hard-switch taxonomy.
- The merge pipeline must produce stable decision artifacts instead of discussion artifacts.

## Proposed Change

- Replace discussion-specific topic data model, heading maps, extraction logic, and rendering templates with decision-specific equivalents.
- Update merge fallback data builders and merge unions to work on decision fields.
- Update CLI merge-content input branching to pass decision-shaped existing context.

## Scope

In scope:

- `emptyTopicDataByCategory`, `extractTopicDataByCategory`, `buildTopicBodyByCategory`, and `topicSignalsFromCategoryData` for decision category.
- `topicDataFromDigestItem`, `topicDataFromMergeContent`, and `mergeTopicData` decision branch updates.
- CLI merge-content input construction branch moved from `discussion` to `decision`.
- File-path/category expectations in merge preparation tests (`topics/decision/...`).

Out of scope:

- Report fallback scanning behavior.
- README and broad CLI argument example rewrites.

## Implementation Notes

- Preserve deterministic ordering, de-duplication, and idempotent merge behavior.
- Keep category-specific section names contract-level and consistent across parse/build functions.
- Avoid introducing compatibility readers for legacy discussion section names.

## Acceptance Criteria

- [ ] Canonical topic parser/renderer no longer contains discussion category branches.
- [ ] Decision topic parser/renderer produces the new decision section structure.
- [ ] Merge fallback builders in `topic-merge.ts` support `decision` category end-to-end.
- [ ] CLI merge-content category branch uses `decision` input shape.
- [ ] `tests/files.test.ts` and `tests/topic-merge-service.test.ts` are updated for decision paths/categories.
