# T002 - Replace prompt-shaped JSON with Structured Outputs

## Problem / Goal

- Digest generation currently relies on prompt instructions to coerce JSON text, then parses free-form model output.
- This is brittle (markdown fences, missing keys, enum drift) and requires defensive parsing logic that Structured Outputs can eliminate.
- Goal: enforce schema-conformant output at the API layer using strict `json_schema` response format.

## Proposed Change

- Extend the OpenAI client to support `response_format` payloads for Chat Completions.
- Define a strict digest schema (object with `items[]`) covering:
  - `category`: `planning | research | discussion`
  - `source`: `slack | wiki | git | figma`
  - `summary`: non-empty string
  - `keyPoints`: string[]
  - `references`: array of `{ source, link }`
- Update digest generation to send schema-based requests instead of “return JSON” prompt coercion.
- Keep runtime validation and clear error messages for API errors/refusals, with fail-fast behavior for unsupported models.

## Scope

In scope:

- API request changes in `src/ai/openai.ts` to pass strict schema response format.
- Digest generation update in `src/digest.ts` to consume structured payload.
- Test updates in `tests/digest.test.ts` and OpenAI client tests.
- README note about model compatibility and fail-fast behavior.

Out of scope:

- Migrating to the Responses API SDK helpers (`responses.parse`, zod helpers).
- Adding non-OpenAI providers.
- Automatic fallback to legacy prompt parsing.

## Non-goals

- Supporting arbitrary user-defined digest schemas.
- Silent model switching when schema format is unsupported.

## Implementation Notes

- Preserve Bun + `fetch` approach; no SDK migration required for this ticket.
- Use strict schema (`additionalProperties: false`, required fields, enum constraints).
- Detect and surface explicit refusals and malformed payloads as user-facing CLI errors.
- Fail fast if model rejects structured response format.

## Acceptance Criteria

- [ ] Digest request uses strict structured output schema instead of prompt-only JSON formatting.
- [ ] Successful responses produce digest items without fenced-JSON extraction heuristics.
- [ ] Unsupported-model/schema errors are explicit and fail fast with actionable message.
- [ ] Existing category/source constraints are enforced by schema + runtime guardrails.
- [ ] Test suite covers success path, refusal/error path, and unsupported-model behavior.
- [ ] README documents structured-output requirement and compatibility expectation.
