# T023 - Merge quality improvements for routing, content relevance, and tags

## Problem / Goal

- Current merge quality is unstable: digest notes can be routed to a partially related topic, and deterministic union merging can introduce duplicated or unrelated content into `Key Points` and `Timeline`.
- Topic routing currently allows multiple targets, which increases duplication risk across topic files for the same digest note.
- Tag handling can proliferate near-duplicate tags over time because merge behavior mostly unions tags rather than explicitly preferring existing canonical tags.
- Goal: improve topic coherence and maintainability by routing each digest item to one primary topic, filtering unrelated merged content, and enforcing stronger tag reuse rules.

## Evidence

- Example quality issue: `tmp/acme/topics/planning/release-cadence-policy-monthly-with-skip-bundling-and-hotfix-rules.md` mixes release-cadence policy decisions with broader release-plan/schedule/task-breakdown content.
- Existing merge implementation in `src/services/topic-merge.ts` performs deterministic list union + dedupe for canonical sections, which cannot reliably detect semantic irrelevance.

## Proposed Change

- Change merge routing contract from multi-target to a single primary `target` per digest item.
- Introduce deterministic candidate pre-ranking and pass only top `N=8` candidates to AI routing to reduce ambiguity and improve matching quality.
- Enrich candidate context in routing prompts with compact canonical previews (not just slug/topic/tags/summary).
- Add a semantic content-merge prompt operation that merges incoming digest content into the selected topic while removing duplicate or unrelated entries.
- Strengthen tag governance: reuse existing category tags where possible and only emit new tags when they add necessary, non-overlapping meaning.
- Preserve robustness with deterministic fallback behavior when semantic merge output is invalid or unavailable.

## Scope

In scope:

- Routing contract changes:
  - Update merge response schema/parsing to single `target` output.
  - Update merge generation function(s) and CLI orchestration to process one target per digest note.
- Candidate ranking and context enrichment:
  - Add deterministic candidate scoring/ranking before routing prompt invocation.
  - Pass top 8 ranked candidates to routing prompt.
  - Extend candidate payload with compact canonical snippets (key points/timeline/references previews).
- Semantic topic merge operation:
  - Add new prompt operation type and built-in prompt definition for topic content merge.
  - Add strict structured-output schema for merged canonical content fields.
  - Integrate semantic merge into merge pipeline with safe fallback to existing deterministic merge logic.
- Tag reuse controls:
  - Build category-level reusable tag pool.
  - Update routing/content-merge prompts with explicit reuse-first guidance.
  - Post-process tags with normalization, dedupe, and bounded growth rules.
- Tests/docs:
  - Update/add tests for single-target routing, ranking behavior, semantic relevance filtering, and tag reuse.
  - Update README merge behavior documentation.

Out of scope:

- Full historical cleanup/migration of existing topic files containing noisy mixed content.
- Any UI/UX redesign of merge confirmation prompts beyond behavior needed for single-target flow.
- Introducing external storage/index services for topic similarity search.

## Non-goals

- Perfect semantic clustering across all categories and languages in one iteration.
- Eliminating all false positives/false negatives in routing decisions.
- Replacing deterministic fallback merge with hard AI dependency.

## Implementation Notes

- Single-target behavior is a product decision to reduce duplication; do not retain optional multi-target output in the default merge path.
- Keep merge idempotency behavior based on digest identity/reference checks; quality improvements must not regress no-op detection.
- Candidate pre-ranking should be deterministic and inexpensive (token overlap and metadata/reference signals) to keep runtime predictable.
- Semantic merge prompt must explicitly forbid importing unrelated schedule/task lines when the selected topic scope is policy-only (and vice versa).
- Tag controls should prioritize existing tags in the same category to prevent drift like `release-plan` vs `release-planning` unless a new tag is clearly required.
- Ensure errors in semantic merge parsing/refusal fall back safely to deterministic merge, with verbose logs indicating fallback usage.

## Acceptance Criteria

- [ ] Merge routing returns one primary target per digest item (single-target schema + parser + runtime behavior).
- [ ] CLI merge applies at most one topic merge plan per pending digest note.
- [ ] Deterministic pre-ranking is implemented and only top 8 candidates are sent to routing prompt.
- [ ] Routing prompt includes richer candidate context and explicit guidance to avoid cross-topic contamination.
- [ ] A semantic content-merge prompt operation is added with strict structured output and integrated into merge flow.
- [ ] Semantic merge removes obvious duplicates/unrelated key points and timeline entries for the selected topic scope.
- [ ] Tag reuse is preferred; new tags are added only when necessary and normalized consistently.
- [ ] Existing idempotency/no-op behavior remains intact for repeated equivalent merges.
- [ ] Test suite covers single-target routing, ranking, semantic relevance filtering, and tag governance behavior.
- [ ] README documents updated merge behavior, including single-target routing and candidate pre-ranking.
