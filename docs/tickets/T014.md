# T014 - Consolidate source_refs and merged_ingest_ids into merged_digest_ids

## Problem / Goal

- Topic file front matter currently stores duplicate ingestion identity data in both `source_refs` and `merged_digest_ids`.
- This duplication increases maintenance cost and causes naming confusion.
- Consolidate to a single canonical field named `merged_digest_ids` while preserving idempotent merge behavior.

## Proposed Change

- Replace dual-field metadata model with one field: `merged_digest_ids`.
- Update topic merge logic to read and write only `merged_digest_ids` as canonical output.
- Keep backward compatibility by reading legacy `source_refs` and `merged_ingest_ids` when present and normalizing them into the new identity format.
- Update docs and tests to reflect the new field name and behavior.

## Scope

In scope:

- Front matter type updates and serialization output updates.
- Merge idempotency checks based on normalized `merged_digest_ids` values.
- Backward-compatible parsing of existing topic files that still contain legacy fields.
- Test updates and docs updates for the renamed canonical field.

Out of scope:

- Bulk rewriting all historical topic files.
- Changes to digest extraction content, routing, or canonical body sections.

## Non-goals

- Introducing a new migration command for repository-wide front matter rewrites.
- Changing topic body structure (`Summary`, `Key Points`, `Timeline`, `References`).

## Implementation Notes

- Canonical front matter output should include `merged_digest_ids` and should not emit `source_refs` or `merged_ingest_ids`.
- Normalization strategy should preserve existing semantics:
  - references become `refs:<source>: <link>` identifiers
  - no-reference digests keep `hash:<sha1>` identifiers
- Idempotency must remain stable across old and new files.

## Acceptance Criteria

- [ ] New topic merges write only `merged_digest_ids` in front matter.
- [ ] Existing files containing `source_refs` and/or `merged_ingest_ids` still dedupe correctly when merged again.
- [ ] `README.md` documents `merged_digest_ids` as the canonical ingestion metadata field.
- [ ] Tests cover canonical output and backward compatibility with legacy metadata fields.
- [ ] No regression in no-op behavior for already-merged digests.
